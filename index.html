The Problem

Knowledge is everywhere — and nowhere

Payment app documents are scattered across 4+ systems. Finding the right answer requires knowing where to look first.

📦 Box

  • Specs & technical docs
  • Budget documents
  • Calculation tables
  • Primary source — no native Onyx connector

📘 Confluence

  • Product documentation
  • Technical design docs
  • Architecture decisions
  • Self-hosted Data Center at rakuten-it.com

☁️ Microsoft 365

  • SharePoint: marketing & sales docs
  • Teams: project discussions
  • Outlook: email threads (v2)
  • Requires Azure AD app registration
🎯

Goal: One chat interface where any team member can ask a question in plain language and get an AI-generated answer with links back to the exact source documents — without moving those documents anywhere.

Solution

Why Onyx?

Open-source, self-hosted, no new cloud subscriptions. Runs entirely on your internal VM.

Two key data flows

📥 Indexing Path

Background worker → fetches docs from connectors → sends to embedding model → writes vectors to OpenSearch + metadata to Postgres

💬 Query Path

Browser → nginx → api_server → hybrid search in OpenSearch → context sent to LLM gateway → streamed answer + citations back to user

🔐 Auth Path

api_server handles sessions against Postgres · AUTH_TYPE=oidc · SSO via Azure AD — same login as Outlook, Teams & SharePoint * Appendix D

Original docs stay in place. Only text embeddings (~few KB per chunk) are stored on the VM. Box/Confluence/SharePoint files are never copied.

⚠️ OpenSearch Memory — Critical

Set heap to exactly 50% of VM RAM. On 32 GB VM: OPENSEARCH_JAVA_OPTS=-Xms8g -Xmx8g. If OpenSearch OOMs, it restarts and may corrupt the index.

🔤 Embedding — Local (no config needed) *

The indexing_model_server runs nomic-embed-text locally. No API key required. Uses ~4 GB RAM after model load. No changes needed for v1.

What this model actually does

It reads a piece of text — a paragraph from a document, or your chat question — and converts it into a list of numbers (called a vector or embedding) that captures the meaning of that text mathematically. Two pieces of text that mean similar things produce numbers that are close together, even if they use completely different words. That's what allows Onyx to find "transaction charges are calculated as follows…" when you ask "what are the payment fees?" — same meaning, different words. It never generates answers — that's the LLM gateway's job. Its only job is meaning → numbers, done entirely locally on your VM.

Every chat query travels this path. No external AI calls — everything stays inside your network.

User

↓
api_server (Docker)
Builds RAG prompt = question + retrieved doc chunks from OpenSearch
↓
LiteLLM (inside Docker)

↓

Bearer token auth · streamed response
↓
Back to User
Streamed answer + source document citations

Appendix B — LLM Gateway

Requirements, Verification & Troubleshooting

✅ What your gateway must support

  • Accept POST /v1/chat/completions endpoint
  • Bearer token authentication via Authorization header
  • OpenAI-compatible request body format
  • Streaming responses (stream: true) — required for chat UX
  • Standard error responses (4xx/5xx with JSON body)
  • Reachable from the Docker VM over internal network

🧪 Pre-flight test — run from the VM on Day 3

curl -s https://llm-gateway.rakuten-it.com/v1/chat/completions \ -H "Authorization: Bearer YOUR_API_KEY" \ -H "Content-Type: application/json" \ -d '{ "model": "YOUR_MODEL_NAME", "messages": [{"role": "user", "content": "Say hello"}], "stream": false }'

Pass: JSON response with choices[0].message.content
Fail: 401 / 403 / 404 — fix before configuring Onyx

⚠️ Common failures & fixes

404 — Model not found
Model name string is wrong. Gateway often uses internal names like rakuten/gpt-4o — confirm exact string with gateway team.
401 — Unauthorized
API key is wrong, expired, or scoped to a different project. Get a fresh key from the gateway team.
Chat hangs / times out
VM firewall blocks outbound to gateway host. Check: curl -v https://llm-gateway.rakuten-it.com from the VM.
Garbled / truncated answers
Gateway streaming format is not fully OpenAI-compatible. Ask gateway team to confirm SSE format compliance.
3 things to confirm with the gateway team before Day 3
1
Exact base URL of the gateway endpoint
2
Exact model name string to use in the "model" field
3
An API key scoped to the payment app team or project

Appendix E — IT Request

Whitelisting HuggingFace — Risks, Reality & How to Ask IT

The VM needs to reach huggingface.co once on first boot to download the embedding model. Here's how to handle it.

Risks IT will flag
Data exfiltration vector
A whitelisted domain could be used to send data out — sets a precedent that the VM can reach the internet.
Malicious models
HuggingFace hosts models from anyone — wrong model name could download something unexpected.
Supply chain attack
ML model files are large binaries — not easily inspectable. A real and documented threat in 2025/2026.
Permanent open door
IT prefers one-time exceptions over permanent firewall rules.
Why the risk is actually low
One-time only
After first boot the model is cached in a Docker volume. The domain is never called again — request a temporary whitelist.
Well-known, audited model
nomic-embed-text is a widely used open-source model from a reputable organisation — not an arbitrary file.
SHA256 verifiable
The downloaded model can be verified against a known hash before use — addresses supply chain concern.
No ongoing traffic
After setup, zero outbound calls to HuggingFace — IT can remove the rule immediately after.
Cleanest alternative — skip IT entirely
Pre-download the model on your own laptop (you have internet via Claude Code gateway), then load it into the Docker volume manually before first start.
No firewall change. No IT ticket. No risk discussion.
Recommended path if IT approval looks slow or difficult.

We need a temporary one-time outbound HTTPS connection from
VM [IP ADDRESS] to the following domains:
