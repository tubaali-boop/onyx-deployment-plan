Onyx Deployment Plan
The Problem

Knowledge is everywhere — and nowhere

Payment app documents are scattered across 4+ systems. Finding the right answer requires knowing where to look first.

📦 Box

  • Specs & technical docs
  • Budget documents
  • Calculation tables
  • Primary source — no native Onyx connector

📘 Confluence

  • Product documentation
  • Technical design docs
  • Architecture decisions
  • Self-hosted Data Center at rakuten-it.com

☁️ Microsoft 365

  • SharePoint: marketing & sales docs
  • Teams: project discussions
  • Outlook: email threads (v2)
  • Requires Azure AD app registration
🎯

Goal: One chat interface where any team member can ask a question in plain language and get an AI-generated answer with links back to the exact source documents — without moving those documents anywhere.

Solution

Why Onyx?

Open-source, self-hosted, no new cloud subscriptions. Runs entirely on your internal VM.

🔒 Fully Internal

Runs on your company VM, accessible over VPN only. Documents never leave your network — only text embeddings are stored locally.

🔌 Native Connectors

Built-in support for Confluence (Data Center), SharePoint, and Teams. No custom integration code for these sources.

🤖 Uses Your LLM Gateway

Points directly at Rakuten's internal OpenAI-compatible LLM endpoint. No external AI API calls, no extra cost.

💬 Chat with Citations

Answers include links back to the exact source document, page, or Teams message. Users can verify and drill in.

🔄 Auto-Sync

Connectors poll on a schedule (every 10–60 min). New and updated documents are automatically re-indexed.

👥 Multi-User

Web UI accessible by the whole team. Username/password auth restricted to rakuten.com domain. 10+ users supported.

Two key data flows

📥 Indexing Path

Background worker → fetches docs from connectors → sends to embedding model → writes vectors to OpenSearch + metadata to Postgres

💬 Query Path

Browser → nginx → api_server → hybrid search in OpenSearch → context sent to LLM gateway → streamed answer + citations back to user

🔐 Auth Path

api_server handles sessions against Postgres · AUTH_TYPE=basic · domain-restricted to rakuten.com / rakuten-it.com

Original docs stay in place. Only text embeddings (~few KB per chunk) are stored on the VM. Box/Confluence/SharePoint files are never copied.

⚠️ OpenSearch Memory — Critical

Set heap to exactly 50% of VM RAM. On 32 GB VM: OPENSEARCH_JAVA_OPTS=-Xms8g -Xmx8g. If OpenSearch OOMs, it restarts and may corrupt the index.

🌐 Outbound Internet Required (once)

The embedding model server downloads ~1.5 GB from HuggingFace on first start. Whitelist huggingface.co on the VM firewall before Day 1.

👤 First Admin Account

Immediately after containers are healthy, open the UI and create the first account. The first account auto-becomes admin. Do this before sharing the URL with the team.

Risk Register

Top Risks & Mitigations

1
OpenSearch OOM crash
Severity: High

2

Severity: High (primary source)
Use Box Drive sync → File connector. Test Box Drive Linux client early. Fallback: rclone.

Severity: High (blocks Day 4)
Identify Rakuten Azure Global Admin before Day 4. Consent is mandatory — connector silently fails without it.
4
Confluence auth confusion (DC vs Cloud)
Severity: Medium
Uncheck "Is Cloud". Use PAT not password. Check logs for 401 errors if sync stalls.
5
SharePoint client secret expires in 24 months
Severity: Medium (silent failure)
Set calendar reminder for 22 months after Day 4. Expiry causes silent indexing stop.
6
Model server slow on first start
Severity: Medium
Wait 10 min before testing. Ensure huggingface.co is reachable from VM on Day 1.
7
LLM gateway model name mismatch
Severity: Medium
Confirm exact model name string with gateway team before Day 3. Test with curl first.
8
Confluence service account missing permissions
Severity: Medium
Use a Confluence admin account for the indexer service account to avoid space-level gaps.

Appendix A — LLM Gateway

How Onyx connects to the Rakuten LLM Gateway

Every chat query travels this path. No external AI calls — everything stays inside your network.

User
Types a question in Onyx chat UI
↓
api_server (Docker)
Builds RAG prompt = question + retrieved doc chunks from OpenSearch
↓
LiteLLM (inside Docker)
Translates request to OpenAI-compatible format, injects API key
↓
Rakuten LLM Gateway (internal network)
POST /v1/chat/completions
Bearer token auth · streamed response
↓
Back to User
Streamed answer + source document citations

FieldExample Value

📡 Actual HTTP request Onyx sends

POST https://llm-gateway.rakuten-it.com/v1/chat/completions Authorization: Bearer sk-rakuten-xxxx Content-Type: application/json { "model": "gpt-4o", "messages": [ {"role": "system", "content": "You are a helpful assistant..."}, {"role": "user", "content": "What are the fee structures?\n\n[Doc chunks...]"} ], "stream": true }
Appendix B — LLM Gateway

Requirements, Verification & Troubleshooting

✅ What your gateway must support

  • Accept POST /v1/chat/completions endpoint
  • Bearer token authentication via Authorization header
  • OpenAI-compatible request body format
  • Streaming responses (stream: true) — required for chat UX
  • Standard error responses (4xx/5xx with JSON body)
  • Reachable from the Docker VM over internal network

🧪 Pre-flight test — run from the VM on Day 3

curl -s https://llm-gateway.rakuten-it.com/v1/chat/completions \ -H "Authorization: Bearer YOUR_API_KEY" \ -H "Content-Type: application/json" \ -d '{ "model": "YOUR_MODEL_NAME", "messages": [{"role": "user", "content": "Say hello"}], "stream": false }'

Pass: JSON response with choices[0].message.content
Fail: 401 / 403 / 404 — fix before configuring Onyx

⚠️ Common failures & fixes

404 — Model not found
Model name string is wrong. Gateway often uses internal names like rakuten/gpt-4o — confirm exact string with gateway team.
401 — Unauthorized
API key is wrong, expired, or scoped to a different project. Get a fresh key from the gateway team.
Chat hangs / times out
VM firewall blocks outbound to gateway host. Check: curl -v https://llm-gateway.rakuten-it.com from the VM.
Garbled / truncated answers
Gateway streaming format is not fully OpenAI-compatible. Ask gateway team to confirm SSE format compliance.
3 things to confirm with the gateway team before Day 3
1
Exact base URL of the gateway endpoint
2
Exact model name string to use in the "model" field
3
An API key scoped to the payment app team or project
Appendix C — Under the Hood (1/2)

The 11 Docker Services — Simply Explained

Think of the whole system as a library building. Each service is one person who works there, doing one specific job.

🚪
nginx
The Front Door Security Guard
Everyone who wants to use Onyx walks through this guard first. He checks you came in through the right door (port 443, HTTPS only), and points you either to the website or the backend. He also holds the lock and key (TLS certificate) so nobody can eavesdrop on conversations.
🖥️
web_server
The Receptionist & Display Board
This is the pretty face — the chat interface you see in your browser. It doesn't do any thinking itself. It just shows you buttons, text boxes, and answers. When you type a question, it passes it to the brain in the back office (api_server).
🧠
api_server
The Head Librarian (The Brain)
The most important service. When you ask a question, this is who figures out the answer. It searches the document index, picks the most relevant chunks, writes a prompt, sends it to the LLM gateway, and streams the reply back to you — with source links.
🔄
background
The Night-Shift Filing Clerk
Works quietly in the background 24/7 without you seeing it. Every 10–60 minutes it checks Box, Confluence, SharePoint, and Teams for new or updated documents, reads them, and files them into the index so the Head Librarian can find them later.
🗄️
relational_db (PostgreSQL)
The Membership Register
A big orderly notebook that stores who the users are, their passwords, their roles, and which connectors are configured. Also keeps track of metadata like "this document came from Confluence, last synced at 3pm." Nothing fancy — just reliable record-keeping.
🔍
opensearch
The Magic Search Shelf
This is where all document content lives as numbers (vectors). When you ask "what are the payment fees?", it doesn't search for exact words — it finds documents that mean the same thing, even if they use different words. Think of it as a very smart librarian who understands meaning, not just spelling.
Appendix C — Under the Hood (2/2)

The 11 Docker Services — Simply Explained

The remaining 5 services — each one still doing its own one job, all day, every day.

📬
cache (Redis)
The Sticky-Note Board
A super-fast scratch pad used for short-lived messages. When the background worker finds new documents to index, it sticks a note here saying "hey, go process this file." Other services grab those notes and act on them. If the power goes off, all notes disappear — that's fine, it'll just write new ones next sync.
📦
minio
The Storage Room
A big storage room for actual files — like if someone uploads a PDF directly into Onyx. It works exactly like Amazon S3 but runs locally on your server. Most of the time you won't notice it exists, but it's there holding files that are too big to fit elsewhere.
🧮
indexing_model_server
The Translator (runs locally)
Takes every document chunk and converts it into numbers (called an embedding) that capture its meaning. This is what makes the search so smart. It runs entirely on your server using nomic-embed-text — no internet call needed, no cost per document. Think of it as a translator turning words into coordinates on a map of meaning.
🎯
inference_model_server
The Result Ranker
After OpenSearch finds 20 possible document chunks, this service re-ranks them — like a judge deciding which ones are truly most relevant to your question. It also runs locally. The api_server then uses only the top-ranked chunks when building the answer, keeping the LLM prompt short and accurate.
🤖
Rakuten LLM Gateway (not a Docker service — your internal API)
The Genius on Speed Dial
This one lives outside Docker — it's Rakuten's internal LLM (like GPT-4o or Claude). The api_server calls it only when someone asks a question. It receives the question + the relevant document chunks, and writes the actual answer in plain English. It's like calling a genius friend who's read everything — you tell them the relevant pages, they summarise the answer for you. All calls stay inside Rakuten's network. No data goes to OpenAI or Anthropic directly.
💡

One-line summary of how they work together: The Filing Clerk (background) reads your docs → the Translator (indexing_model_server) converts them to numbers → stored in the Magic Shelf (opensearch). When you ask a question → the Brain (api_server) searches the shelf → the Ranker picks the best bits → the Genius (LLM gateway) writes the answer → the Receptionist (web_server) shows it to you.
